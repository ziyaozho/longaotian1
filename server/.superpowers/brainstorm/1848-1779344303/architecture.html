双模式架构设计

Demo 模式如何无缝融入现有代码，不破坏正常游戏逻辑

核心原理

模式切换：URL 参数控制
https://game.example.com/?demo=true&seed=demo_v1

无需构建两份产物，同一份代码根据 URL 参数切换行为

架构分层

数据流：从上到下，Demo 层在 AI 层和编排层介入
🖥️ UI 层 — GameMain.tsx, DemoOverlay.tsx [新增]
↓ props / store
🎮 编排层 — orchestrator.ts, demoOrchestrator.ts [新增]
↓ 根据 isDemo 选择路径
🤖 AI 层 — deepseek.ts
Demo 模式
→ precacheService.ts [新增]
正常模式
→ deepseek API
↓
📦 数据层 — stores (playerStore, gameStore, narrativeStore) — 不改动
↓
🗂️ 内容层 — scenes.ts, systems.ts, achievements.ts, precache/ [新增]

新增文件清单

demoConfig.ts

Demo 模式配置：预设角色属性、世界选择、系统绑定、事件序列时间线、旁白文案

demoOrchestrator.ts

Demo 流程编排器：按时间线推进，在每个阶段触发对应的 AI 内容、UI 动画、旁白提示

precacheService.ts

预缓存服务：管理预生成的 AI 内容，Demo 模式下秒出，同时检查缓存有效性

DemoOverlay.tsx

演示叠加层：旁白提示条、阶段指示器、倒计时、当前亮点标注、快捷键说明

关键设计决策

1

isDemo 通过 React Context 传递

不在 store 中持久化 demo 状态。从 URL 参数读取 → 存入 Context → 全局可用。避免 demo 状态被意外保存到 localStorage。

2

orchestrator 在入口处分叉，不在每个函数中判断

只在 turn() 的入口做一次 isDemo 判断，然后委托给 demoOrchestrator 或正常 orchestrator。避免在 10 个函数中散落 isDemo 检查。

3

预缓存内容放在独立 JSON 文件中，不混入现有 data/

precache/ 目录独立管理，方便后续用脚本批量重新生成缓存内容。Git 中标记为 generated content。
