# 演示流程与完整 API 接入设计方案

> 项目：人生重开模拟器 - AI + 系统流
> 比赛：抖音 AI 创变者计划 · 互动空间赛道
> 日期：2026-05-21

## 一、背景与目标

### 比赛约束
- 演示时间：3-5 分钟（标准路演）
- 赛道方向：AI 驱动的交互式体验
- 交付形式：H5 Web 应用（抖音小程序 WebView 备选）
- 演示重点：AI 实时生成 (40%) + 叙事深度 (30%) + 玩法系统 (30%)

### 技术栈现状
- React 19 + TypeScript + Vite + Zustand + Framer Motion + Tailwind CSS
- DeepSeek API 已对接，带本地模板回退
- 5 基础世界 + 3 隐藏世界，10 个系统，30+ 成就
- 完整的"万界行者"叙事系统

## 二、方案概述：双模式架构

在现有游戏基础上增加 Demo 模式开关，通过 URL 参数 `?demo=true` 激活。Demo 模式和正常模式共享同一套代码，仅在 AI 层和编排层分叉。

### 设计原则
- **最小改动**：只改 3 个现有文件，其余全部新增
- **零分叉**：不维护两套构建产物
- **透明回退**：Demo 模式任何异常自动回退到正常流程

## 四、架构设计

### 数据流
```
UI 层 (GameMain.tsx, DemoOverlay.tsx)
  ↓
编排层 (orchestrator.ts, demoOrchestrator.ts)  ← isDemo 分叉点
  ↓
AI 层 (deepseek.ts)
  ├── Demo 模式 → precacheService.ts → precache/*.json
  └── 正常模式 → DeepSeek API
  ↓
数据层 (stores) — 不改动
  ↓
内容层 (scenes, systems, achievements, precache/)
```

### 新增文件
| 文件 | 职责 |
|------|------|
| `src/demo/demoConfig.ts` | Demo 配置：预设角色、世界、系统、事件序列、旁白文案 |
| `src/demo/demoOrchestrator.ts` | Demo 流程编排：按时间线推进阶段 |
| `src/demo/precacheService.ts` | 预缓存管理：读取、验证、回退 |
| `src/demo/DemoOverlay.tsx` | 演示叠加层：旁白提示、计时器、快捷键 |
| `src/demo/precache/demo_v1.json` | 预生成的 AI 内容缓存 |

### 改动文件
| 文件 | 改动量 | 说明 |
|------|--------|------|
| `orchestrator.ts` | +15 行 | turn() 入口处判断 isDemo |
| `contentGenerator.ts` | +10 行 | generateScene() 入口处分叉 |
| `GameMain.tsx` | +5 行 | 条件渲染 DemoOverlay |

### 关键决策
- `isDemo` 通过 React Context 传递，不持久化到 Store
- 编排层只在入口处分叉一次，不在各函数中散落检查
- 预缓存内容独立 `precache/` 目录，可被 `.gitattributes` 标记为 generated

## 五、预缓存系统

### 生命周期
1. **构建时**：`pnpm run precache` 脚本读取 demoConfig 事件序列，逐条调用 DeepSeek API，结果写入 `precache/demo_v1.json`
2. **运行时（Demo）**：precacheService 直接从 JSON 读取，配合打字机动画伪装实时感
3. **运行时（正常）**：完全不受影响

### 缓存结构
```json
{
  "version": "demo_v1",
  "generatedAt": "2026-05-21T10:00:00Z",
  "events": {
    "char_create": { "sceneText": "...", "choices": [...], "systemMessage": "..." },
    "turn_1": { "...": "..." },
    "climax_reveal": { "...": "..." }
  }
}
```

约 10-15 个事件节点，总缓存体积约 15-25KB。

### 容错
- 缓存未命中 → 自动回退实时 API
- 版本不匹配 → 构建时报错
- Demo 模式下优先缓存，不走网络

## 七、完整 API 接入方案

### 流式响应（SSE）
- DeepSeek API `stream: true`
- SSE 事件流 → 逐 token 渲染到打字机组件
- 目标：首字延迟 < 500ms

### 上下文窗口管理
| 组成部分 | Token 预算 | 说明 |
|----------|-----------|------|
| 系统提示词 | ~500 | 世界观 + 世界规则 + 系统人格 + 输出格式 |
| 角色状态 | ~200 | 6 维属性 + HP/MP + 装备 + 任务 + 成就 |
| 滑动窗口历史 | ~1500 | 最近 5 轮的选择+结果摘要，超出的做渐进摘要压缩 |
| 当前上下文 | ~300 | 当前场景 + 上一轮选择 + 事件类型 |

总计约 2500 tokens/请求，DeepSeek 128K 上下文窗口余量充足。

### 3 级容错体系
1. **Level 1**：DeepSeek API 流式响应（正常路径）
2. **Level 2**：非流式重试，3 次指数退避（网络波动/超时）
3. **Level 3**：本地模板引擎（完全离线，300+ 手写场景）

### 成本估算
- 每局约 50 回合 × 3000 tokens = 150K tokens ≈ ¥0.15
- Demo 模式：预缓存，0 API 调用
- 离线模式：本地模板，0 API 调用

## 八、部署方案

### 主力：腾讯云 COS 静态网站托管
- 国内完全可用，无需翻墙
- 开启 CDN 加速全国节点
- 新用户有 50GB/月免费额度
- 4 步部署：创建存储桶 → 构建上传 → 绑定域名 → CDN 加速

### 备用：CloudFlare Pages
- 部分国内节点可达
- Git Push 自动部署
- 国际访问作为备份

### 兜底：抖音小程序 WebView
- 如果比赛要求小程序形态
- 创建小程序壳工程 + WebView 加载 H5
- JSBridge 适配层（约 30 行）
- 游戏核心零改动

### 构建优化
- 代码分割：precache 文件单独 chunk
- 移动端适配：viewport 优化、触摸支持、安全区域
- Service Worker：离线可用
- 二维码：部署后自动生成
